# Collective Data Export & Import

## Context

Users of the Harmonic-hosted instance should be able to export an entire collective's data and re-import it into a separate (e.g., self-hosted) instance. This is the first phase of the [data lifecycle management](data-lifecycle-management.md) roadmap. We build export and import simultaneously so each validates the other.

Scope: full collective export — all content, participants, votes, attachments, audit chains, links. User-level export comes later.

## Core Design Principle

**Round-trip fidelity**: `export(collective) |> import(target_instance) |> export(re-imported) ≈ original export`. The export format is the canonical representation of a collective's data.

## User Identity Strategy

On import, users are matched by email:
- **Match found**: Map to existing user in target instance
- **No match**: Create a **placeholder user** (`user_type: "imported_placeholder"`) with display name and email from export, but no auth credentials. Placeholder users appear in history/votes but cannot log in. They can be "claimed" later by a real user who proves ownership (e.g., via email verification).

## Export Format

Single ZIP file containing JSON. One file per model type, plus attachments as binary files.

```
harmonic-collective-export/
  manifest.json                         # metadata, version, checksums
  collective.json                       # collective record + settings
  users.json                            # all referenced users (email, name, handle, type)
  members.json                          # collective memberships + roles
  notes.json                            # includes comments and statements
  decisions.json
  options.json
  decision_participants.json
  votes.json
  decision_audit_entries.json           # full hash chain, preserved as-is
  commitments.json
  commitment_participants.json
  links.json
  note_history_events.json
  invites.json                          # active invites only
  heartbeats.json
  representation_sessions.json          # collective-scoped only
  representation_session_events.json
  attachments.json                      # metadata
  attachments/                          # binary files
    <uuid>-<filename>
```

### manifest.json

```json
{
  "format_version": "1.0",
  "app_version": "1.14.0",
  "exported_at": "2026-05-07T12:00:00Z",
  "source_instance": "app.harmonic.team",
  "collective": {
    "name": "Engineering",
    "handle": "engineering"
  },
  "record_counts": {
    "notes": 42,
    "decisions": 7,
    "commitments": 3
  },
  "checksums": {
    "notes.json": "sha256:abc123...",
    "decisions.json": "sha256:def456..."
  }
}
```

### ID Handling

Export preserves original UUIDs as `source_id` fields. On import, new UUIDs are generated for all records. The import service maintains a mapping table (`source_id -> new_id`) to remap all foreign keys.

### Excluded

- **Events** (system audit log — regenerated by tracked callbacks on import)
- **SearchIndex / CycleDataRow** (materialized views — rebuilt automatically)
- **UserItemStatus** (ephemeral UI state — not collective-scoped)
- **ChatSessions / ChatMessages** (separate collective type, not part of standard export)
- **AutomationRules / AutomationRuleRuns / AutomationRuleRunResources** (TBD — may reference external webhooks/agents not present in target; needs design)
- **WebhookDeliveries** (automation delivery log — excluded with automation)
- **AiAgentTaskRuns / AiAgentTaskRunResources** (agent execution log — system data)
- **ContentReports** (moderation state, not user content)
- **TrusteeGrants** (cross-user delegation — not collective-scoped)
- **Notifications / NotificationRecipients** (user-specific, not collective data)
- **Links** (regenerated automatically from text content via `Linkable` after_save callback; exported for reference but skipped on import. Soft-deleted items lose backlinks — acceptable since content is scrubbed)

### Data model audit note

The database has 43 tables with `collective_id` (or `resource_collective_id`). This plan accounts for all of them — 16 model types are exported, the rest are explicitly excluded above. Several tables have nullable `collective_id` (options, votes, note_history_events, etc.) — these derive their collective from parent records and are queried via the parent FK (e.g., `Vote.where(decision_id:)`) rather than directly by `collective_id`.

## Architecture

### Export

```
CollectiveExportService
  - Queries all collective-scoped models
  - Serializes each to JSON (custom export format, not api_json)
  - Downloads attachment blobs to temp dir
  - Computes SHA-256 checksums per file
  - Writes manifest.json
  - Creates ZIP
  - Attaches to DataExport record via ActiveStorage
```

### Import

```
CollectiveImportService
  - Validates manifest (format version, checksums)
  - Extracts ZIP to temp dir
  - Phase 1: Create/match users (build user mapping)
  - Phase 2: Create collective + members
  - Phase 3: Create content (notes, decisions, commitments) using ID mapping
  - Phase 4: Create relationships (options, votes, participants, links)
  - Phase 5: Create activity records (history events, audit entries)
  - Phase 6: Attach files
  - All in a single transaction (rollback on any failure)
```

### ID Mapping

The import service maintains an in-memory hash:

```ruby
@id_map = {}  # { "source_uuid" => "new_uuid" }

def map_id(source_id)
  return nil if source_id.nil?
  @id_map.fetch(source_id) { raise "Unmapped ID: #{source_id}" }
end
```

Every record is created with a new UUID, and `@id_map[source_id] = new_record.id` is recorded immediately after creation. Subsequent records use `map_id` to resolve FKs.

## Text Link Rewriting on Import

Notes, decisions, and commitments contain internal links in their text content. The `LinkParser` detects two formats:
- Full URLs: `https://subdomain.hostname/collectives/handle/n/abc12345`
- Markdown paths: `[text](/collectives/handle/n/abc12345)`

These reference items by **truncated_id** (first 8 chars of UUID) and **collective handle**.

On import, text content must be rewritten because:
1. **Truncated IDs change** — new UUIDs mean new truncated_ids
2. **Collective handle may change** — target collective could have a different handle
3. **Hostname changes** — different instance means different base URL

**Strategy**: After all records are created (and ID mapping is complete), run a text rewriting pass:
1. Build a `source_truncated_id → new_truncated_id` mapping from the ID map
2. For each Note/Decision/Commitment text field, find internal link patterns using the same regex as `LinkParser`
3. Replace source truncated_ids with new truncated_ids, source collective handle with new handle, source hostname with target hostname
4. Save the updated text (this triggers `parse_and_create_link_records!` via the `Linkable` after_save callback, which automatically creates the correct Link records)

This means **Link records do not need to be imported directly** — they are regenerated from the rewritten text. Export them anyway for completeness/debugging, but import skips them.

## Representation Sessions with Cross-Collective Resources

`RepresentationSessionEvent` records can reference resources (`resource_type`/`resource_id`) that belong to a **different collective** (via `resource_collective_id`). On import:

- If the referenced resource is within the same collective being imported → remap the ID normally
- If the referenced resource is in a **different collective** (not part of this export) → set `resource_type` and `resource_id` to nil, which causes the app to use the `DeletedRecordProxy` pattern (returns `[deleted]` for display)

Same approach for `context_resource_type`/`context_resource_id` on events.

## Audit Chain Handling

Decisions created before `AUDIT_CHAIN_LAUNCH_DATE` (2026-05-05) have no audit chain — the app already supports this via `decision.audit_chain_enabled?` which checks `created_at >= AUDIT_CHAIN_LAUNCH_DATE`.

For **imported decisions with audit chains**, the same pattern applies:
- Audit entries are exported as-is (the historical chain from the source instance)
- On import, set the decision's `created_at` to its **original** value (preserving history)
- But the hash chain will NOT re-verify because actor IDs and other inputs have changed
- **Mark imported decisions as pre-audit-chain**: Set `audit_chain_hash` to nil. The verifier already handles this — when `audit_chain_hash` is nil, the chain-hash-mismatch check is skipped
- Exported audit entries are stored as **read-only historical records** with a metadata flag indicating they are imported
- Any **new** actions on the imported decision (new votes, closing, etc.) will start a fresh audit chain if `audit_chain_enabled?` returns true based on the preserved `created_at`

This avoids needing a separate "imported chain" concept — we reuse the existing pre-launch-date handling.

## Rate Limiting

- **Application-level**: 1 concurrent export per collective
- **Rack::Attack**: 3 export requests/hour per user

## Cleanup

`CleanupExpiredExportsJob < SystemJob` — daily, purges exports where `expires_at < Time.current`.

## Testing Strategy

The key insight: **export → import → re-export → diff** proves correctness.

1. **Unit: `CollectiveExportService`** — Create collective with known data, export, verify ZIP structure and JSON content
2. **Unit: `CollectiveImportService`** — Import a known export ZIP, verify all records created with correct relationships
3. **Round-trip test** — Export collective A, import into fresh tenant, re-export, compare JSON (ignoring IDs and timestamps)
4. **User mapping test** — Import with some matching users and some new, verify placeholder creation
5. **Attachment test** — Export with attachments, import, verify files are accessible
6. **Error handling** — Corrupt ZIP, missing checksums, invalid format version, duplicate import
7. **Controller tests** — Admin-only access, rate limiting, download flow
