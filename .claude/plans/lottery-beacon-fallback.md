# Lottery Beacon Fallback

## Context

When the external randomness beacon (drand) is unreachable or returns unverifiable results, a closed lottery gets stuck in a "Drawing..." state with no way to resolve it. The creator has no feedback and no options.

## Idea

Provide a **fallback randomness source** that uses internally generated randomness (e.g., `SecureRandom`). When the external beacon fails, present the lottery creator with a choice:

1. **Retry external beacon** — try fetching from drand again
2. **Use internal randomness** — draw the lottery immediately using server-generated randomness, clearly marked as not externally verifiable

The decision and verify page should clearly indicate which source was used. Internally-drawn lotteries would show a notice explaining that the randomness was generated by the server and is not independently verifiable.

## Key Considerations

- The creator makes the choice, not the system — the fallback is never automatic
- The UI needs to communicate the tradeoff clearly without being alarming
- The verify page should adapt: externally-drawn lotteries show full verification; internally-drawn lotteries explain why verification isn't available
- The `RandomnessProvider` interface may need a concept of "verifiable vs unverifiable" so views can adapt
- Need to handle the job failure path: after N retries, stop retrying and surface the failure to the creator instead of silently giving up
